# Agent OS Decisions

이 문서는 Agent OS의 현재 확정된 운영 결정에 대한 단일 진실 공급원이다. 역사적 기획서나 체크리스트와 내용이 충돌하면 이 문서를 우선한다. 아직 확정되지 않은 항목은 상태를 `파일럿` 또는 `후보`로 표시한다.

## D-002 실행 권한과 격리

- 상태: 확정
- 날짜: 2026-07-15

### 결정

- Lab은 폐기 가능한 격리 환경 내부에서 풀액세스로 실행한다.
- Project는 저장소 내부에서 자율 실행하고, 시스템 권한 승인은 반복해서 요청하지 않는다.
- Production은 읽기와 검증 중심으로 운영하고 외부 상태 변경은 승인받는다.
- 이번 작업에서 만든 임시 파일, 빌드 산출물, 캐시는 승인 없이 삭제할 수 있다.
- 작업 전부터 존재한 파일·디렉터리, Git 미추적 파일, 영구 데이터의 삭제·초기화는 실행 직전에 사용자에게 확인한다.
- 코드와 문서 내부의 일반적인 추가·수정·부분 제거는 파일 삭제 승인의 대상이 아니다.
- 저장소 루트, 대량 파일, 백업 없는 데이터의 삭제는 자동 실행하지 않는다.

### 이유

에이전트의 자율성을 확보하면서 데이터의 존재 자체를 없애는 행위만 별도로 통제하기 위해서다. 명령어 이름보다 복구 가능성과 데이터 손실 가능성을 기준으로 판단한다.

### 결과

- Lab에는 호스트 파일, 자격증명, 사설 네트워크를 노출하지 않는다.
- push, PR, 배포, 메시지, 결제는 승인 대상이다.
- 시스템 샌드박스는 자율 실행 범위를 제한하고, 삭제 확인은 Agent OS 행동 정책으로 보완한다.
- 기본 Project 설정은 `workspace-write`, `approval_policy = "never"`, Developer OS 볼트 쓰기와 네트워크 허용으로 구성한다.
- Project는 Git, Lab은 폐기와 재생성을 기본 복구 수단으로 사용한다.

## D-003 기록 계층과 자동 수집

- 날짜: 2026-07-15

### 결정

- 원시 세션은 `~/.codex/sessions`에 둔다.

### 결과

- 이벤트 큐는 30일 보존을 기본값으로 한다.
- Maintainer가 작업 5건 또는 7일을 초기 검토 주기로 사용한다.
- 이벤트 수집 실패는 본 작업을 중단시키지 않는다.

## D-004 모델과 추론 라우팅

- 상태: 파일럿
- 날짜: 2026-07-15

### 결정

- 기본 실행은 `gpt-5.6-sol`의 `medium` 추론을 사용한다.
- 독립적이고 경계가 명확한 저위험 보조 작업은 `fast_worker`가 `gpt-5.6-terra`의 `low` 추론으로 수행한다.
- 고영향 설계, 보안, 데이터 손실 위험, 표준 시도 실패 후 복잡한 진단, 최종 독립 검토는 `critical_reviewer`가 `gpt-5.6-sol`의 `high` 추론으로 검토한다.
- T0와 단일 단계 작업은 위임하지 않는다. T1도 전달 비용보다 절감 효과가 명확할 때만 위임한다.
- 동시 하위 에이전트는 최대 3개, 중첩 깊이는 1로 제한한다.

### 이유

항상 높은 추론을 사용하지 않으면서도 위험하거나 되돌리기 어려운 판단의 품질을 유지하기 위해서다. 위임 자체의 컨텍스트 복제 비용도 측정 대상에 포함한다.

### 검증

- 첫 10개 실제 작업에서 모델, 추론 수준, 위임 수, 관측 토큰 또는 크레딧, 성공 여부와 재작업을 기록한다.
- 품질 저하나 재시도로 총비용이 증가하면 해당 라우팅을 되돌린다.

## D-005 비용 계측과 작업 흐름

- 상태: 파일럿
- 날짜: 2026-07-15

### 결정

- Stop Hook이 세션 누적 토큰과 마지막 모델 요청 토큰을 로컬 이벤트 큐에 자동 기록한다.
- 작업 간 사용량은 같은 세션의 연속 Stop 누적치 차이로 계산한다.
- ChatGPT 크레딧, API 과금, 관측 토큰은 서로 다른 지표로 저장하며 임의 환산하지 않는다.
- 가역적인 작업은 중간 승인을 받지 않고 완료하고, 이해 확인은 마지막 튜토리얼·퀴즈·인터뷰로 묶는다.
- 영구 삭제, 되돌릴 수 없는 외부 행동, 검증된 복구 경로가 없는 변경만 실행 전에 확인한다.

### 한계

- `last_request`는 전체 턴이 아니라 마지막 모델 요청 한 건이다.
- 로컬 세션 토큰만으로 ChatGPT 크레딧이나 API 비용을 계산할 수 없다.
- 첫 Stop 이전에는 작업별 차이를 계산할 기준점이 없다.

## D-006 Developer OS 복구 계층

- 상태: 확정
- 날짜: 2026-07-15

### 결정

- Developer OS 볼트는 로컬 Git 저장소로 관리한다.
- macOS LaunchAgent가 변경 사항을 매시간 로컬 커밋하며 외부로 push하지 않는다.
- Codex는 보호된 `.git`에 직접 쓰지 않고, 별도 프로세스가 스냅샷을 담당한다.
- 로컬 Git은 파일 복구 이력일 뿐 장치 손실에 대한 백업으로 간주하지 않는다.

### 오프디바이스 경로

- 주 오프디바이스 경로는 비공개 Git 원격으로 선택했다.
- 매시간 로컬 Git 스냅샷은 유지하고, Obsidian Git의 자동 push는 사용하지 않으며 검토 후 수동으로 pull·push한다.
- `miniminjae92/developer-os` 비공개 원격을 별도 임시 위치에 clone하고 핵심 문서와 Git 이력을 확인해 복구 경로를 검증했다.

## D-007 책임 기반 Tutor와 학습 깊이

- 상태: 파일럿
- 날짜: 2026-07-18
- 적용: 사용자가 기존 완료 후 일괄 설명보다 나을 것으로 판단해 활성 Tutor 스킬 적용을 승인했다.

### 결정

- Tutor의 학습 범위는 CEO, CTO, 개발자 같은 직함이 아니라 사용자가 소유한 결정, 검증할 결과, 감당할 실패로 정한다.
- 이해 깊이는 `L0` 용어와 지도, `L1` 작동 원리, `L2` 선택지와 트레이드오프, `L3` 검증·실패·복구, `L4` 직접 구현·디버깅으로 구분한다.
- 필수 이해는 사용자 책임, 실패 영향, 비가역성이 클수록 깊게 한다.
- 에이전트에 실행을 위임한 작업은 모든 구현 세부보다 의사결정과 검증에 필요한 충분한 깊이를 우선한다. 다른 적격 검토자 없이 사용자가 승인해야 하면 더 깊게 설명한다.
- 학습은 현재 결정에 필요한 `필수`, 반복·전이 가치가 있는 `전략적`, 시간이 있을 때 탐구할 `호기심`으로 구분한다.
- 가역적인 실행은 수업 때문에 중단하지 않는다. 중요한 결정 지점이나 작업 완료 후 필요한 깊이만 설명하고, 전략적·호기심 주제는 후속 후보로 남긴다.
- 사용자가 명시적으로 공부를 요청하면 현재 위험 수준을 넘어 더 깊게 학습할 수 있다.

### 검증

- 서로 다른 실제 작업 5건에서 목표 깊이, 설명으로 인한 실행 중단, 결정·검증·복구의 재설명 가능성, 후속 학습 후보의 재사용을 기록한다.
- 저위험 일회성 작업은 과잉 설명 없이 완료되고, 고위험 또는 핵심 결정은 최소 `L3`까지 다뤄지는지 확인한다.
- 파일럿 뒤 별도 학습 큐나 자동화가 필요한지 결정하며, 그전에는 Run의 후속 학습 후보를 사용한다.

### 롤백

- 정책이 과도한 설명이나 기록 부담을 만들면 Tutor를 기존의 완료 후 핵심 결정·복구 설명 방식으로 되돌린다.
- 기존 Run은 변경하지 않으며 새 형식은 선택적 본문 구역이므로 제거해도 기록 원본은 손상되지 않는다.

## D-008 공급자 중립 지시사항 계층

- 상태: 확정
- 날짜: 2026-07-21

### 결정

- 지시사항의 단일 원본은 공급자 중립 코어(`agents/AGENTS.md`)에 둔다.
- Codex와 Gemini는 각자의 컨텍스트 파일을 코어에 심링크한다. 어댑터 파일을 두지 않는다.
- Claude는 `CLAUDE.md`에서 `@` import로 코어를 읽고, 그 아래 얇은 Claude 전용 구역만 유지한다. Claude Code는 AGENTS.md를 네이티브로 읽지 않고, 심링크는 어댑터를 놓을 자리를 없애기 때문이다.
- 항상 참인 지시만 코어에 둔다. 조건부로 참인 지시(특정 워크플로, 포맷)는 전부 스킬로 내보낸다.
- 코어의 상한은 50줄이다. 초과하면 스킬로 분리한다. 코어는 세 CLI가 매 세션 무조건 지불하는 컨텍스트 비용이기 때문이다.

### 이유

지시사항이 공급자별로 갈라지면 같은 작업의 에이전트 행동이 CLI에 따라 달라져 라우팅 실험의 비교 기준이 무너진다. 변환·생성 파이프라인 대신 심링크와 import를 쓰는 것은 어댑터를 얇게 유지하기 위해서다.

## D-009 구독 자원 라우팅

- 상태: 파일럿
- 날짜: 2026-07-21

### 결정

- 라우팅 선언은 `agents/routing.json`에 두고 `planner/worker/reviewer/mechanical` 논리 역할과 provider→account→model→effort 4축으로 관리한다. D-004의 Codex 내부 라우팅을 공급자 전체로 확장한 것이다.
- Claude Max의 5시간 윈도우는 부패 자원으로 취급해 Sonnet으로 채우고, 주간 캡은 저장 자원으로 취급해 Opus 사용만 배급한다.
- Codex 2계정은 예비가 아니라 오버플로우 밸브다. 윈도우·주간 캡 소진 시 작업을 멈추지 않고 Codex로 흘린다.
- 기계적 작업(커밋 메시지, 분류, 요약)은 Gemini Flash 정적 기본값을 유지한다. 윈도우 잔량 기반 동적 전환은 레버가 없는 최적화이므로 만들지 않는다.
- 강결합 편집과 연속 설계 판단은 한 세션에서 모델 교체 없이 수행한다. 정확도를 깨뜨리는 것은 모델 교체가 아니라 컨텍스트 단절이다.
- routing.json에서 claude_* 참조를 제거해도 시스템이 성립해야 한다(구독 중단 내성).

### 검증

- 에스컬레이션 로그(D-010)와 주간 리뷰로 라우팅 오류를 집계하고, 품질 저하나 재작업 증가 시 해당 역할의 기본값을 되돌린다.

## D-010 에스컬레이션 이벤트 로깅

- 상태: 파일럿
- 날짜: 2026-07-21

### 결정

- 하위 티어에서 시작했다가 실패해 상위 티어로 재시도한 작업(에스컬레이션)과, 상위 티어에서 한 번에 끝난 기계적 작업(하향 후보)을 기록한다.
- 기록은 기존 `agent-os-friction --origin agent` 채널에 `escalation: <from>→<to> | <작업> | <이유>` 형식으로 남기고, 주간 리뷰가 집계해 routing.json 변경을 제안한다.
- 사후 토큰 회계는 라우팅 개선의 간접 신호이므로 정밀 측정 체계를 만들지 않는다. 에스컬레이션이 라우팅 오류의 유일한 직접 증거다.

### 이유

새 수집 인프라 없이 기존 마찰 기록 채널에 규약만 얹어, 라우팅 변경 제안이 감이 아니라 근거에서 나오게 한다.
