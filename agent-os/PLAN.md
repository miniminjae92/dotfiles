## 1. 핵심 결정

Agent OS v0.1의 목표는 **AI 작업의 성공 여부와 비용을 일관되게 기록하고 변경의 순효과를 판단하는 최소 운영체계**를 만드는 것이다.

초기에는 다음을 하지 않는다.

- 별도 또는 중첩 Git 저장소 생성
- Workspace Agent와 ChatGPT 프로젝트 우선 구축
- 모든 작업에 목표 문서와 반복 평가 강제
- 여러 모델, 프롬프트, 도구를 동시에 변경
- 상시 로드되는 대형 지침 추가
- 효과가 확인되지 않은 스킬과 자동화 생성

`agent-os/`는 현재 닷파일 저장소 안에 두며, 검증된 자산만 기존 `.codex/skills/`와 `install.sh` 패턴으로 연결한다.

## 2. 해결할 문제

1. 작업별 성공률, 시간, 토큰 및 비용을 비교하기 어렵다.
2. ChatGPT 크레딧, API 과금, 컨텍스트 사용량이 혼용된다.
3. 지침과 도구가 누적되면서 토큰과 유지비가 증가한다.
4. baseline과 반복 평가가 개선 효과보다 더 비쌀 수 있다.
5. 결과가 좋아도 재현 조건이 기록되지 않는다.

운영 순서는 `관측 → 제한 → 비교 → 결정 → 필요한 경우 자산화`로 고정한다.

## 3. v0.1 완료 조건

- 실제 작업 10건의 실행 기록이 있다.
- 8건 이상에서 사용량 출처를 확인할 수 있다.
- 최소 한 작업에서 baseline과 candidate를 비교했다.
- 성공, 품질, 시간, 사용량을 개별 지표로 비교했다.
- 예산 초과 실행은 중단되었거나 승인 기록이 있다.
- 변경 하나에 대해 채택 또는 폐기 결정을 내렸다.
- 실행 기록 작성 시간의 중앙값이 5분 이하다.
- 안전 게이트 위반이 없다.

v0.1의 성공은 토큰 절감 자체가 아니라 **품질과 토큰의 교환 관계를 측정하고 통제할 수 있는 상태**다.

## 4. 운영 원칙

### 4.1 Budget before Execution

T1과 T2 작업은 실행 전에 성공 기준, 비용 상한, 중단 조건을 정한다.

### 4.2 Cheapest Sufficient Path

- 결정적 테스트가 있으면 LLM 평가를 추가하지 않는다.
- 단일 에이전트로 가능한 작업에 서브에이전트를 사용하지 않는다.
- 짧은 작업에 전체 템플릿을 적용하지 않는다.
- 추가 검색과 파일 읽기는 성공 기준에 필요할 때만 수행한다.

### 4.3 Quality Is a Gate

토큰이 적더라도 핵심 성공 기준을 통과하지 못하면 개선이 아니다.

### 4.4 Reuse Valid Baselines

실행 조건이 같으면 기존 baseline을 재사용한다. 비교할 때마다 다시 실행하지 않는다.

### 4.5 Evidence before Asset

한 번 성공한 결과를 바로 스킬이나 에이전트로 만들지 않는다.

### 4.6 Reversible and Local First

초기 변경은 현재 저장소에서 되돌릴 수 있게 관리한다. 외부 쓰기와 권한 확대는 별도 승인을 요구한다.

## 5. 비용 회계

### 5.1 과금 모드

| `billing_mode` | 의미 | 기록 단위 |
| --- | --- | --- |
| `chatgpt_credit` | ChatGPT 로그인 기반 Work 또는 Codex | 크레딧·계정 사용량, 관측 토큰 |
| `api_metered` | API 키 기반 실행 | 토큰, 실제 API 비용 |
| `local_unmetered` | 로컬 모델 | 시간, 로컬 자원, 관측 토큰 |
| `unknown` | 계측 불가 | 미측정으로 기록 |

ChatGPT 크레딧을 API 달러 비용으로 환산하지 않는다. 서로 다른 과금 모드의 값을 직접 합치거나 비교하지 않는다.

### 5.2 필수 지표

- `accepted_success`: 사용자 검토 후 실제 채택 여부
- `total_tokens` 또는 `credits_used`
- `cost_usd`: API에서 실제 확인되는 경우만 기록
- `elapsed_minutes`
- `human_review_minutes`
- `tool_calls`, `retries`, `subagent_count`
- `avoidable_interventions`: 실패나 잘못된 가정으로 발생한 사용자 개입

측정할 수 없는 값은 0이 아니라 `null`로 기록한다. 안전상 필요한 승인 요청은 불필요한 사용자 개입으로 세지 않는다.

핵심 경제성 지표는 `채택된 성공 1건당 사용량`이다. Agent OS가 추가한 기록, 평가, baseline 실행 비용도 포함한다.

## 6. 작업 등급과 초기 예산

| 등급 | 예시 | 생성 토큰 파일럿 상한 | 도구 호출 | 재시도 | 서브에이전트 | 기록 |
| --- | --- | ---: | ---: | ---: | ---: | --- |
| T0 | 짧은 질문, 단일 명령, 오탈자 수정 | 3,000 | 3 | 1 | 0 | 예외만 |
| T1 | 단일 파일 수정, 일반 리뷰, 문서 작성 | 20,000 | 20 | 2 | 원칙상 0 | 간단 기록 |
| T2 | 다중 파일 변경, 고위험 분석, 복합 조사 | 60,000 | 50 | 2 | 최대 2 | 전체 기록 |

이 값은 목표가 아니라 초기 중단 기준이다. 세션 JSONL의 누적 토큰 차이를 작업 구간의 관측값으로 사용하고, 컨텍스트 사용률, 도구 호출, 재시도 및 `/usage`의 계정 지표를 함께 본다.

입력 토큰에는 캐시된 장기 컨텍스트가 포함될 수 있으므로 총 토큰만으로 작업 난이도나 ChatGPT 크레딧을 판단하지 않는다. 입력, 캐시 입력, 출력, 추론 출력, 계정 크레딧을 분리해 기록한다.

생성 토큰은 `output_tokens`와 `reasoning_output_tokens`의 관측값을 함께 보는 파일럿 지표다. 모델·클라이언트별 포함 관계가 달라지면 중복 합산하지 않고 원시 필드를 각각 비교한다.

T0와 T1은 가장 낮은 충분 추론 수준에서 시작한다. 높은 추론 수준과 서브에이전트는 T2 또는 실패 근거가 있을 때만 사용한다.

### 예산 체크포인트

- 60%: 성공 기준 대비 진척을 확인하고 불필요한 탐색과 위임을 중단한다.
- 80%: 새 범위를 추가하지 않고 핵심 구현·검증·복구에만 사용한다.
- 100%: 안전한 롤백을 제외한 실행을 중단하고 완료·미완료·다음 최소 단계를 기록한다.

Stop 이벤트는 턴 완료 후 기록되므로 이 규칙은 강제 차단기가 아니라 실행 루프의 중간 점검 기준이다.

- 60%: 새로운 탐색 경로를 열지 않는다.
- 80%: 현재 해법 수렴과 필수 검증을 우선한다.
- 100%: 중단하고 상태, 실패 원인, 재개 비용을 보고한다.

상한 확대, 새로운 접근법, 서브에이전트 추가는 사용자 승인을 받는다.

### 컨텍스트 제한

- 컨텍스트 70% 도달 전에 정리하거나 새 세션으로 분리한다.
- 긴 로그와 전체 파일을 반복 삽입하지 않는다.
- 오래된 중간 결과는 요약하되 결정, 제약, 검증 결과는 보존한다.
- 전체 Agent OS 문서 대신 필요한 정책과 템플릿만 참조한다.

## 7. 실행 루프

1. **Classify:** T0, T1, T2 등급을 정한다.
2. **Define:** 목표, 성공 기준, 제약, 예산, 중단 조건을 정한다.
3. **Reuse:** 기존 자산과 유효한 baseline을 찾는다.
4. **Execute:** 가장 저렴한 충분 경로로 실행한다.
5. **Verify:** 테스트, 성공 기준, 사용자 검토 순서로 검증한다.
6. **Record:** 결과, 비용, 실패, 사용자 수정만 간결하게 기록한다.
7. **Decide:** Adopt, Conditional, Iterate, Reject, Observe 중 하나를 선택한다.

검증 우선순위는 `결정적 테스트 → 성공 기준 → 사용자 검토 → 필요한 경우에만 LLM 평가`다.

## 8. Baseline 및 반복 정책

다음이 같거나 비교 목적상 동등하면 baseline을 재사용한다.

- 작업과 입력 버전
- 성공 기준과 평가 rubric
- 모델과 추론 수준
- 핵심 도구와 실행 환경
- 관련 지침 버전

작업 입력, 모델 계열, 평가 방식 또는 결과에 직접 영향을 주는 환경이 바뀌면 baseline을 다시 확보한다.

반복 횟수는 고정하지 않는다.

- 결정적 테스트가 있는 구현: 기본 1회
- 출력 편차가 예상되는 작업: 1회 후 판정이 경계일 때 2회 추가
- 고위험 채택 결정: 근거가 있을 때 최대 5회

추가 실행이 결론을 바꿀 가능성이 없으면 반복하지 않는다.

## 9. 변경 채택 기준

### 공통 게이트

- 핵심 성공 기준 100% 통과
- 안전 게이트 통과
- 확인된 핵심 회귀 없음
- 실행 조건과 사용량 출처 기록
- 롤백 가능
- 예산 초과 시 승인 기록 존재

### 품질 프로필

같은 과금 모드와 측정 단위 안에서 비교한다.

| 프로필 | 초기 채택 기준 |
| --- | --- |
| Economy | 품질 비열등, 채택 성공당 사용량 20% 이상 감소 |
| Balanced | 품질 또는 성공률 개선, 사용량 증가 10% 이내 |
| Quality | 중요한 품질 개선, 사용량 증가 50% 이내 및 사유 승인 |

비율은 첫 10건 이후 보정한다. 종합점수 하나로 비용이나 실패를 숨기지 않는다.

### 자산화 기준

스킬, 프롬프트, 에이전트 또는 자동화로 승격하려면 다음을 모두 만족해야 한다.

- 비슷한 작업에서 3회 이상 사용
- 2회 이상 실제 결과 채택
- 순시간 또는 비용 절감 확인
- 유지보수 책임과 삭제 조건 정의

충족하지 못하면 실행 기록이나 예시로만 보존한다.

## 10. 안전 및 데이터 경계

| 등급 | 예시 | 처리 원칙 |
| --- | --- | --- |
| Public | 공개 문서와 저장소 | 허용 |
| Internal | 개인 닷파일과 비공개 문서 | 승인된 계정과 범위에서 허용 |
| Sensitive | 개인정보와 회사 비공개 자료 | 최소 범위, 명시적 승인 |
| Secret | 비밀번호, API 키, 세션 토큰 | 입력·출력·저장 금지 |

클라우드 모델에 파일 내용을 제공하는 것도 외부 처리로 본다.

다음이 발생하면 결과를 채택하지 않는다.

- 비밀정보 노출 또는 저장
- 승인 없는 외부 메시지, 게시, 결제, 배포
- 승인 없는 파괴적 변경 또는 권한 확대
- 사용자 변경 덮어쓰기
- 복구 불가능한 변경
- 평가 결과 조작 또는 실패 은폐

작업 전부터 존재한 파일·디렉터리나 영구 데이터의 삭제·초기화는 파괴적 변경으로 본다. 반면 이번 작업에서 생성한 임시 산출물의 정리와 코드·문서 내부의 일반적인 편집은 별도 삭제 승인 없이 수행한다.

## 13. 단계별 도입

### Phase 0: 정책 확정

- 이 문서의 작업 등급, 초기 상한, 데이터 경계를 검토한다.
- v0.1과 충돌하는 실행 지침은 v0.2를 우선하도록 명시한다.
- 현재 모델과 추론 수준을 baseline에 기록하고 첫 측정 중에는 동시에 변경하지 않는다.

완료 조건: 비용 단위와 초기 예산에 대한 사용자 승인.

### Phase 1: 측정 MVP

- 최소 `agent-os/` 구조를 만든다.
- 앞으로 발생하는 실제 작업 10건을 기록한다.
- 별도 벤치마크용 작업을 인위적으로 만들지 않는다.

완료 조건: v0.1 완료 조건 중 실행 기록 관련 항목 충족.

### Phase 2: 첫 비교 실험

- 반복 작업 하나와 유효한 baseline을 선택한다.
- 프롬프트, 모델, 도구 중 한 요소만 변경한다.
- 품질 프로필을 선택하고 candidate를 평가한다.
- 평가 비용을 포함해 채택 여부를 결정한다.

완료 조건: 결과 원인과 순효과를 설명할 수 있음.

### Phase 3: 선택적 자산화

- 채택된 변경을 실제 작업에서 2회 이상 추가 사용한다.
- 자산화 기준을 통과할 때만 스킬 또는 Workspace Agent를 만든다.
- 사용하지 않는 자산은 30일 후 삭제 후보로 지정한다.

ChatGPT 프로젝트, Workspace Agent, 앱 연결, 반복 평가 자동화는 Phase 3 이전에 필수로 취급하지 않는다.

## 14. 계측 및 운영

Codex에서는 지원되는 범위에서 다음을 사용한다.

- `/status`: 모델, 컨텍스트 사용량, 세션 설정
- `/usage`: 계정 사용량
- `/statusline`: 컨텍스트와 토큰 항목 표시
- OTel `response.completed`: 자동 수집이 필요할 때의 토큰 수

OTel을 활성화해도 사용자 프롬프트 본문은 기본 로깅하지 않는다. 실험적인 rollout budget 기능은 v0.1의 핵심 통제로 의존하지 않는다.

10개 작업마다 다음을 검토한다.

- 등급별 사용량과 성공률
- 초기 상한 조정 필요성
- 반복 실패와 불필요한 지침
- 사용하지 않는 자산과 중복 규칙
- Agent OS 기록 및 평가 자체의 비용

다음 조건이면 절차를 축소하거나 폐기한다.

- 기록 시간이 작업 시간의 10%를 지속적으로 초과
- 평가 비용이 예상 절감보다 큼
- 규칙이 개입과 지연을 늘리지만 성공률은 개선하지 않음
- 생성 자산이 30일 동안 사용되지 않음

## 15. 최초 실행 순서

1. 이 문서를 검토하고 Git에 기록한다.
2. `agent-os/` 최소 구조만 만든다.
3. 앞으로 발생하는 실제 작업 10건을 기록한다.
4. 작업 등급과 토큰 상한을 보정한다.
5. 가장 반복적인 작업 하나를 첫 비교 대상으로 선택한다.
6. 변경 요소 하나만 실험한다.
7. 효과가 확인된 뒤에만 스킬 또는 Workspace Agent를 검토한다.

지금 만들 대상은 메타 에이전트가 아니라 **측정 가능한 운영 습관**이다.
