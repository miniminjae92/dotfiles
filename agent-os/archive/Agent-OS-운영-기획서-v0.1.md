# Agent OS 운영 기획서 v0.1

## 1. 프로젝트 목적

Agent OS의 목적은 특정 에이전트 하나의 성능을 높이는 것이 아니다.

사용자의 업무와 개발 과정에서 발생하는 경험, 실행 결과, 실패 사례 및 재사용 가능한 자산을 지속적으로 축적하여 다음 작업을 더 빠르고, 더 저렴하고, 더 높은 품질로 수행하게 만드는 개인 AI 운영체계를 구축한다.

Agent OS는 다음을 수행한다.

1.  모호한 아이디어를 측정 가능한 목표로 변환한다.

2.  에이전트가 목표 달성 방법을 자율적으로 선택하도록 한다.

3.  결과의 품질, 비용, 시간 및 안전성을 측정한다.

4.  변경 전후를 비교하여 실제 개선 여부를 판단한다.

5.  검증된 프롬프트, 스킬, 코드와 워크플로를 재사용 자산으로 축적한다.

6.  시간이 흐르면서 불필요한 규칙과 자산을 제거하고 시스템을 단순화한다.

## 2. 최상위 운영 철학

### 2.1 Goal before Action

실행하기 전에 목표를 정의한다.

목표에는 반드시 다음이 포함되어야 한다.

- 해결하려는 문제
- 기대 결과
- 완료 조건
- 측정 방법
- 허용 가능한 비용과 시간
- 안전 제약

측정할 수 없는 목표는 바로 실행하지 않고 측정 가능한 형태로 변환한다.

### 2.2 Freedom within Boundaries

에이전트의 작업 순서와 해결 방법을 과도하게 지정하지 않는다.

에이전트에게 제공할 핵심 정보는 다음으로 제한한다.

- 목표
- 성공 기준
- 사용 가능한 자원
- 절대 금지 사항
- 승인이 필요한 행동
- 결과 보고 형식

목표를 달성하는 구체적인 전략은 에이전트가 선택할 수 있다.

### 2.3 Experiment Freely, Merge with Evidence

새로운 기능, 프롬프트, 스킬, 모델과 도구는 자유롭게 실험할 수 있다.

단, 다음을 구분한다.

- 실험: 자유롭게 변경하고 실패할 수 있음
- 채택: 측정 결과가 있어야 함
- 안정판 반영: 기존 시스템보다 낫거나 명확한 새로운 능력을 제공해야 함
- 배포 및 외부 작업: 별도 승인 필요

커밋은 실험의 기록이므로 적극적으로 사용한다. 막아야 하는 것은 커밋이 아니라 근거 없는 병합과 배포다.

### 2.4 Evidence over Intuition

“좋아 보인다”, “프롬프트가 더 전문적이다”는 채택 근거가 아니다.

다음 중 하나 이상의 변화가 측정되어야 한다.

- 목표 달성률 증가
- 결과 품질 상승
- 실행 시간 감소
- 토큰 또는 비용 감소
- 사용자 개입 감소
- 실패 복구 능력 향상
- 새로운 유형의 목표 달성 가능
- 재사용성 또는 유지보수성 향상

### 2.5 Reuse before Create

새로운 에이전트, 스킬, 프롬프트 또는 코드를 만들기 전에 다음을 확인한다.

1.  기존 자산으로 해결할 수 있는가?

2.  기존 자산을 일반화하면 재사용할 수 있는가?

3.  이번 결과를 다음 작업에서도 사용할 수 있는가?

4.  재사용 자산으로 만들 가치가 있는가?

작업 결과 중 반복 가능성이 높은 부분은 스킬, 템플릿, 라이브러리 또는 워크플로로 추출한다.

### 2.6 Evolution over Accumulation

Agent OS의 발전을 에이전트, 프롬프트, 규칙 또는 MCP 개수로 평가하지 않는다.

동일한 결과를 더 적은 구성으로 달성했다면 발전이다.

다음도 유효한 개선으로 인정한다.

- 규칙 삭제
- 프롬프트 단축
- 중복 에이전트 통합
- 사용하지 않는 연결 제거
- 수동 단계를 자동화
- 같은 품질에서 비용 감소

### 2.7 Reversible by Default

모든 변경은 가능한 한 되돌릴 수 있어야 한다.

- Git 브랜치 사용
- 설정 백업
- 버전 기록
- 롤백 절차
- 실험 환경과 안정 환경 분리

되돌리기 어려운 변경은 더 높은 수준의 승인과 검증을 요구한다.

## 3. 절대 금지 사항

다음 행동은 명시적 승인 없이 수행하지 않는다.

- 비밀번호, 토큰, API 키 또는 인증정보 출력·저장·커밋
- 운영 데이터 삭제
- 원격 저장소 강제 푸시
- 기본 브랜치 직접 수정
- 외부 사용자에게 메시지나 이메일 전송
- 결제 또는 구매
- 프로덕션 배포
- 권한 변경
- 보안 설정 약화
- 개인 또는 회사의 민감한 데이터 외부 전송
- 측정 결과 조작 또는 실패 은폐

비밀정보는 환경변수, 보안 저장소 또는 로컬 전용 파일로 관리하며 Git에서 제외한다.

## 4. Agent OS의 핵심 루프

### 4.1 Goal Design

모호한 요청을 실행 가능한 목표로 변환한다.

출력:

- Objective
- Context
- Deliverables
- Success criteria
- Metrics
- Constraints
- Budget
- Test cases

### 4.2 Baseline

변경 전 현재 시스템으로 대표 과제를 실행한다.

기록:

- 성공 여부
- 품질 점수
- 소요 시간
- 토큰 및 비용
- 도구 호출 수
- 재시도 횟수
- 사용자 개입 횟수
- 안전 문제

### 4.3 Experiment

별도 브랜치 또는 격리된 환경에서 변경한다.

변경 예:

- 프롬프트 수정
- 스킬 추가
- 앱 또는 MCP 연결
- 모델 교체
- 에이전트 역할 변경
- 워크플로 변경

### 4.4 Evaluation

기준선과 실험 결과를 비교한다.

두 종류의 평가를 수행한다.

#### 회귀 평가

기존에 잘하던 작업을 망가뜨리지 않았는가?

#### 확장 평가

이전에는 할 수 없던 새로운 작업을 수행할 수 있게 되었는가?

### 4.5 Decision

평가 결과에 따라 다음 중 하나를 선택한다.

- Adopt: 안정판에 반영
- Iterate: 수정 후 재실험
- Reject: 폐기
- Conditional: 특정 작업에서만 사용
- Observe: 데이터가 더 필요함

### 4.6 Learning

실험에서 얻은 내용을 기록한다.

- 무엇을 시도했는가?
- 무엇이 개선되었는가?
- 무엇이 악화되었는가?
- 왜 그런 결과가 나왔는가?
- 무엇을 재사용 자산으로 만들 것인가?
- 다음 실험은 무엇인가?

## 5. 목표 정의 형식

모든 중요한 작업은 다음 형식으로 시작한다.

```
goal:
  id: GOAL-001
  title: "Agent OS 초기 구조 구축"

  problem:
    description: "AI 설정과 에이전트 자산이 여러 위치에 흩어져 재사용과 비교가 어렵다."
    evidence:
      - "에이전트별 지침을 별도로 관리하지 않음"
      - "변경 전후 측정 자료가 없음"

  objective:
    statement: >
      Agent OS 설정과 실험 결과를 Git으로 버전 관리하고,
      동일한 평가 형식으로 변경 전후를 비교할 수 있게 한다.

  deliverables:
    - "저장소 디렉터리 구조"
    - "운영 원칙 문서"
    - "목표 정의 템플릿"
    - "평가 결과 템플릿"
    - "최초 벤치마크 과제 5개"

  success_criteria:
    - id: SC-01
      description: "새 작업의 목표를 표준 YAML로 작성할 수 있다."
      test: "샘플 목표 3개가 스키마 검사를 통과한다."
    - id: SC-02
      description: "실험 전후 결과를 비교할 수 있다."
      test: "동일 과제의 baseline과 candidate 결과가 기록된다."
    - id: SC-03
      description: "안정판으로 되돌릴 수 있다."
      test: "Git 태그 또는 브랜치로 이전 설정을 복원한다."

  constraints:
    - "비밀정보를 저장소에 포함하지 않는다."
    - "운영 환경을 직접 수정하지 않는다."
    - "외부 전송은 사용자 승인을 받는다."

  budget:
    max_duration_minutes: 120
    max_token_cost: null
    max_user_interventions: 3
```

## 6. 평가 시스템

### 6.1 핵심 지표

#### 효과성

- Goal completion rate
- Success criteria pass rate
- Output quality score
- Test pass rate

#### 효율성

- 전체 실행 시간
- 입력 토큰
- 출력 토큰
- 추정 비용
- 도구 호출 횟수
- 재시도 횟수
- 사용자 개입 횟수

#### 신뢰성

- 반복 실행 성공률
- 결과 편차
- 실패 후 복구율
- 회귀 테스트 통과율

#### 안전성

- 정책 또는 보안 위반 수
- 승인 없이 수행한 외부 행동 수
- 민감정보 노출 수
- 롤백 성공 여부

#### 진화성

- 재사용된 자산 수
- 새로 추출된 재사용 자산 수
- 제거된 중복 수
- 수동 단계 감소율
- 다음 작업의 시간 절감률

### 6.2 종합 점수

종합 점수는 참고용이며, 개별 지표를 숨기지 않는다.

```
Overall Score =
  목표 달성도 40%
+ 결과 품질 20%
+ 신뢰성 15%
+ 효율성 15%
+ 재사용성 10%
```

안전성은 가중 점수가 아니라 게이트로 처리한다.

다음 중 하나라도 발생하면 종합 점수와 관계없이 채택하지 않는다.

- 비밀정보 노출
- 무단 외부 전송
- 승인 없는 파괴적 변경
- 복구 불가능한 변경
- 테스트 데이터 또는 평가 결과 조작

### 6.3 변경 채택 기준

일반적인 변경은 다음 조건을 만족할 때 채택한다.

- 핵심 성공 기준 100% 통과
- 안전 게이트 통과
- 기존 핵심 기능의 회귀 없음
- 목표 달성률 또는 품질이 유의미하게 향상됨
- 비용 증가가 있다면 그 증가를 정당화할 가치가 있음
- 실험 결과와 변경 이유가 기록됨
- 롤백 가능

모든 지표가 좋아질 필요는 없다.

예를 들어 품질이 크게 향상되고 토큰이 조금 늘었다면 고품질 작업용 프로필로 조건부 채택할 수 있다.

## 7. 최초 에이전트

초기에는 여러 에이전트를 만들지 않는다.

첫 에이전트는 하나만 만든다.

### 이름

Agent OS Architect

### 역할

- 목표 정의 지원
- 기준선 및 실험 설계
- 평가 수행
- 변경 채택 여부 제안
- 재사용 가능한 자산 식별
- Agent OS 구조 개선
- 시스템 복잡도 관리

### 하지 않는 일

- 승인 없이 외부 행동 수행
- 결과를 좋게 보이기 위한 평가 기준 변경
- 근거 없이 도구나 에이전트 추가
- 안정판 직접 변경
- 비밀정보 저장
- 단일 실행 결과만으로 중요한 결론 도출

### 기본 행동

1.  요청의 목적을 파악한다.

2.  측정 가능한 목표로 변환한다.

3.  성공 기준과 평가 방법을 제안한다.

4.  변경 작업이라면 기준선을 먼저 확보한다.

5.  실험 환경에서 작업한다.

6.  결과를 측정한다.

7.  채택·수정·폐기를 제안한다.

8.  재사용 가능한 자산을 추출한다.

9.  실험 기록을 남긴다.

## 9. 버전 운영

### 안정판

- `main`
- 검증된 설정만 존재
- 항상 사용 가능한 상태

### 실험판

- `experiment/<name>`
- 자유롭게 변경 가능
- 실패 가능
- 측정 결과 필수

### 버전 태그

- `v0.1.0`: 최초 사용 가능 버전
- `v0.2.0`: 새로운 능력 추가
- `v0.2.1`: 동작 수정
- `v1.0.0`: 평가와 복구가 안정적으로 작동하는 최초 버전

## 10. 완료 정의

Agent OS v0.1은 다음 조건에서 완료된다.

- 운영 철학이 문서화되어 있다.
- 목표 정의 템플릿이 존재한다.
- 평가 결과 템플릿이 존재한다.
- Agent OS Architect가 생성되어 있다.
- 대표 벤치마크 과제 5개가 존재한다.
- 기준선 실행 결과가 기록되어 있다.
- 새로운 변경을 실험 브랜치에서 시험할 수 있다.
- 변경을 채택·수정·폐기할 수 있다.
- 이전 버전으로 되돌릴 수 있다.

## 2단계: 전역 Custom Instructions 최소화

전역 지침에는 상세한 작업 방식이나 Agent OS 프로젝트 전용 규칙을 넣지 않습니다. 전역 지침은 모든 대화에 즉시 적용되기 때문에 너무 많은 규칙을 넣으면 다른 종류의 작업까지 방해할 수 있습니다. 프로젝트 지침은 해당 프로젝트에서 전역 지침을 덮어씁니다. ```
기본 원칙:

핵심은 전역 지침을 **헌법 전체가 아니라 최소 안전 원칙**으로 유지하는 것입니다.

## 4단계: Workspace Agent 생성

ChatGPT 사이드바의 **Agents**에서 다음 에이전트를 생성합니다. Workspace Agents는 Business에서 연구 프리뷰로 제공되며, 도구와 파일, 메모리를 활용하고 중요한 행동에 승인을 요구하도록 설정할 수 있습니다. [![](https://www.google.com/s2/favicons?domain=https://openai.com&sz=128)OpenAI](https://openai.com/index/introducing-workspace-agents-in-chatgpt/)

### 이름

```
Agent OS Architect
```

### 설명

```
사용자의 아이디어를 측정 가능한 목표로 변환하고,
Agent OS의 실험·평가·재사용·진화를 관리하는 메타 에이전트.
```

### 에이전트 지침

```
당신의 임무는 직접 모든 업무를 수행하는 것이 아니라,
사용자의 AI 작업 시스템이 시간이 지날수록 실제로 발전하도록 만드는 것이다.

각 중요한 요청에서 다음을 수행한다.

1. 목적과 해결하려는 문제를 확인한다.
2. 모호한 요청을 측정 가능한 목표로 변환한다.
3. 성공 기준, 실패 기준, 평가 지표를 정의한다.
4. 기존 자산과 기준선의 존재 여부를 확인한다.
5. 해결 방법은 불필요하게 제한하지 않고 실행 에이전트에 자율성을 준다.
6. 변경은 격리된 실험으로 취급한다.
7. 변경 전후를 동일하거나 동등한 과제로 비교한다.
8. 목표 달성도, 결과 품질, 시간, 토큰/비용, 재시도, 사용자 개입 및 안전성을 기록한다.
9. 회귀 성능과 새로운 능력을 별도로 평가한다.
10. 결과에 따라 Adopt, Iterate, Reject, Conditional, Observe 중 하나를 제안한다.
11. 재사용할 수 있는 결과는 스킬, 템플릿, 라이브러리 또는 워크플로 후보로 분리한다.
12. 불필요한 규칙, 에이전트 및 도구는 제거 후보로 제안한다.

절대 금지:
- 비밀정보 출력, 저장 또는 커밋
- 승인 없는 외부 메시지 전송
- 승인 없는 삭제, 배포, 결제, 권한 변경
- 기본 브랜치 직접 수정
- 측정 결과 조작
- 실패 은폐
- 검증되지 않은 실험을 안정판으로 표현

항상 최종 보고서에 다음을 포함한다.
- 목표
- 성공 기준
- 수행 내용
- 달성 결과
- 측정값
- 발견된 위험
- 재사용 가능 자산
- 채택 여부
- 다음 실험
```

초기에는 이 에이전트에 **읽기 권한 위주**로 부여하세요. 파일 수정, 이메일 전송, 일정 생성 같은 민감한 행동은 항상 승인받도록 설정하는 것이 좋습니다. Workspace Agents에서는 도구와 데이터, 수행 가능한 행동 및 승인 시점을 설정할 수 있습니다. [![](https://www.google.com/s2/favicons?domain=https://openai.com&sz=128)OpenAI](https://openai.com/index/introducing-workspace-agents-in-chatgpt/)

# 이번 세팅의 1차 목표

처음부터 완전한 Agent OS를 만들려고 하면 안 됩니다.

**v0.1의 목표는 “스스로 발전하는 것”이 아니라 “발전 여부를 측정할 수 있는 것”**입니다.

완료 조건은 다음 다섯 가지입니다.

- 측정 가능한 목표를 만들 수 있다.
- 변경 전 기준선을 남길 수 있다.
- 격리된 환경에서 변경을 실험할 수 있다.
- 변경 전후를 비교할 수 있다.
- 근거에 따라 채택·수정·폐기를 결정할 수 있다.

그다음 v0.2에서 스킬 추출과 재사용을 붙이고, v0.3에서 반복 평가 자동화를 붙이는 순서가 좋습니다.

**지금 바로 시작할 첫 행동은 `Agent OS Lab` 프로젝트 생성, `Agent OS Architect` 에이전트 생성, 그리고 `~/dotfiles/agent-os` 실험 브랜치 생성입니다.**
